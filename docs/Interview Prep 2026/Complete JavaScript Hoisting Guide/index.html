What Hoisting Actually Is‚Äã

Hoisting ‚â† moving code to the top

This is the most common misconception. JavaScript doesn't physically move your code. Instead, it processes your code in two distinct phases:

The Truth About Hoisting‚Äã

During the creation phase, the JavaScript engine scans through your code and:

  • Registers all variable and function declarations
  • Allocates memory for them
  • Initializes them according to specific rules (depends on declaration type)

During the execution phase, the code runs line by line with those pre-registered bindings already in place.

The Two-Phase Mechanism‚Äã

Phase 1: Creation (Memory Allocation)‚Äã

// What you write:
console.log(x);
var x = 5;

// How JS processes it:
// CREATION PHASE:
var x = undefined;  // Memory allocated, initialized to undefined

// EXECUTION PHASE:
console.log(x);     // undefined
x = 5;              // Assignment happens

Creation phase behavior by type:

Declaration TypeCreatedInitializedValue
var‚úÖ‚úÖundefined
let‚úÖ‚ùåuninitialized (TDZ)
const‚úÖ‚ùåuninitialized (TDZ)
Function declaration‚úÖ‚úÖEntire function
Function expressionVariable only‚ùåundefined (if var)
Arrow functionVariable only‚ùåundefined (if var)
Class‚úÖ‚ùåuninitialized (TDZ)

Phase 2: Execution‚Äã

Code runs line by line, assignments happen, functions are called.

Hoisting Rules Reference Table‚Äã

let‚úÖ‚ùå‚ùå (ReferenceError - TDZ)Block

Temporal Dead Zone (TDZ)‚Äã

The TDZ is the time between entering a scope and the actual declaration line where a variable exists but cannot be accessed.

Basic TDZ Example‚Äã

console.log(a); // ‚ùå ReferenceError: Cannot access 'a' before initialization
let a = 10;

The variable a exists in memory but is in the TDZ until line 2 executes.

TDZ in Block Scope‚Äã

{
  // TDZ starts here for 'x'
  console.log(x); // ‚ùå ReferenceError
  let x = 1;      // TDZ ends here
}

TDZ with Functions‚Äã

function test() {
  // TDZ starts for 'data'
  console.log(data); // ‚ùå ReferenceError
  let data = 100;    // TDZ ends
}
test();

TDZ is Temporal, Not Spatial‚Äã

function useX() {
  console.log(x); // ‚ùå ReferenceError
}

let x = 10;
useX(); // This call itself is fine, but the function body accesses x before declaration

Wait, this actually works because by the time useX() is called, x is already declared. The TDZ only exists during the execution of the scope where the variable is declared.

Correct example:

let x = 10;

function useX() {
  console.log(x); // ‚úÖ 10 - works fine
}

useX();

TDZ matters when accessing before declaration in the same scope:

function useX() {
  console.log(x); // ‚ùå ReferenceError - TDZ
  let x = 10;     // Declaration in same scope
}

useX();

typeof in TDZ‚Äã

console.log(typeof a); // ‚ùå ReferenceError
let a = 10;

But with undeclared variables:

console.log(typeof undeclaredVariable); // ‚úÖ "undefined"

üëâ Key insight: typeof does NOT bypass TDZ for declared variables.

var Hoisting‚Äã

var declarations are hoisted and initialized to undefined.

Basic var Hoisting‚Äã

console.log(x); // undefined
var x = 5;
console.log(x); // 5

Behind the scenes:

var x = undefined;  // Creation phase
console.log(x);     // undefined
x = 5;              // Execution phase
console.log(x);     // 5

var in Functions‚Äã

function test() {
  console.log(a); // undefined
  var a = 10;
  console.log(a); // 10
}
test();

var Scope (Function-scoped)‚Äã

function example() {
  if (true) {
    var x = 10;
  }
  console.log(x); // ‚úÖ 10 - var ignores block scope
}
example();

Multiple var Declarations‚Äã

var a = 1;
var a = 2;
console.log(a); // 2 - allowed, no error

var in Global Scope‚Äã

var globalVar = 'test';
console.log(window.globalVar); // 'test' (in browsers)

let and const Hoisting‚Äã

Both let and const are hoisted but not initialized, creating a TDZ.

let Hoisting‚Äã

console.log(x); // ‚ùå ReferenceError
let x = 10;

const Hoisting‚Äã

console.log(y); // ‚ùå ReferenceError
const y = 20;

Block Scope‚Äã

{
  let x = 10;
  const y = 20;
}
console.log(x); // ‚ùå ReferenceError - not defined outside block
console.log(y); // ‚ùå ReferenceError

No Re-declaration‚Äã

let a = 1;
let a = 2; // ‚ùå SyntaxError: Identifier 'a' has already been declared
const b = 1;
const b = 2; // ‚ùå SyntaxError

const Requires Initialization‚Äã

const x; // ‚ùå SyntaxError: Missing initializer in const declaration

Function Declaration Hoisting‚Äã

Function declarations are fully hoisted and initialized.

Basic Function Hoisting‚Äã

sayHello(); // ‚úÖ "Hello!"

function sayHello() {
  console.log("Hello!");
}

Behind the scenes:

// Creation phase:
function sayHello() {
  console.log("Hello!");
}

// Execution phase:
sayHello(); // ‚úÖ "Hello!"

Function Overriding‚Äã

foo(); // "second"

function foo() {
  console.log("first");
}

function foo() {
  console.log("second");
}

Last declaration wins during creation phase.

Function Expression Hoisting‚Äã

Function expressions only hoist the variable, not the function.

var Function Expression‚Äã

sayHi(); // ‚ùå TypeError: sayHi is not a function

var sayHi = function() {
  console.log("Hi!");
};

Why TypeError, not ReferenceError?

// Behind the scenes:
var sayHi = undefined;  // Hoisted
sayHi();                // undefined is not a function
sayHi = function() {    // Assignment happens after
  console.log("Hi!");
};

let/const Function Expression‚Äã

greet(); // ‚ùå ReferenceError: Cannot access 'greet' before initialization

const greet = function() {
  console.log("Greetings!");
};

Arrow Function Hoisting‚Äã

Arrow functions behave like function expressions.

const Arrow Function‚Äã

sayHi(); // ‚ùå ReferenceError

const sayHi = () => {
  console.log("Hi!");
};

var Arrow Function‚Äã

sayHi(); // ‚ùå TypeError: sayHi is not a function

var sayHi = () => {
  console.log("Hi!");
};

Class Hoisting‚Äã

Classes are hoisted but remain in TDZ until declaration.

Basic Class Hoisting‚Äã

const obj = new Person(); // ‚ùå ReferenceError

class Person {
  constructor(name) {
    this.name = name;
  }
}

Class Expression‚Äã

const obj = new MyClass(); // ‚ùå ReferenceError

const MyClass = class {
  constructor() {}
};

Interview Problems‚Äã

üî• Problem 1: var in Function Scope‚Äã

function test() {
  console.log(a);
  var a = 10;
}
test();

Output: undefined

Explanation: var a is hoisted to the top of the function and initialized to undefined.

üî• Problem 2: let vs var‚Äã

console.log(a);
var a = 1;

console.log(b);
let b = 2;

Output:

undefined
ReferenceError: Cannot access 'b' before initialization

üî• Problem 3: Function Declaration vs var‚Äã

foo();

function foo() {
  console.log("function");
}

var foo = function() {
  console.log("var");
};

Output: "function"

Explanation:

// Creation phase:
function foo() { console.log("function"); } // Function fully hoisted
var foo = undefined; // var is also hoisted but doesn't override function

// Execution phase:
foo(); // Calls the function
var foo = function() { console.log("var"); }; // Reassignment happens after

üî• Problem 4: Function in Block (Strict vs Non-Strict)‚Äã

"use strict";
{
  function foo() {
    return "inside block";
  }
}
foo(); // ‚ùå ReferenceError in strict mode

Non-strict mode (browser-dependent):

{
  function foo() {
    return "inside block";
  }
}
foo(); // May work in some browsers

üëâ Best practice: Never use function declarations inside blocks. Use function expressions instead.

üî• Problem 5: Arrow Function Hoisting‚Äã

  console.log("hi");

üî• Problem 6: Class Hoisting‚Äã

üî• Problem 7: Shadowing‚Äã

var a = 1;

function test() {
  console.log(a);
  var a = 2;
}

test();

Output: undefined

Explanation: Local var a shadows the outer a and is hoisted to the top of the function.

üî• Problem 8: Function Parameters‚Äã

function foo(a) {
  console.log(a);
  var a = 20;
}

foo(10);

Output: 10

Explanation: Parameter a = 10 is already initialized. The var a = 20 declaration is ignored (parameter takes precedence), but the assignment a = 20 would happen after the console.log.

üî• Problem 9: Duplicate Declarations‚Äã

var a = 1;
var a = 2;
console.log(a); // 2 - allowed

let b = 1;

üî• Problem 10: Loop Hoisting‚Äã

for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}

Output:

3
3
3

Explanation: var i is function-scoped (or global if not in a function), so there's only one binding shared across all iterations.

Fix with let:

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Output: 0, 1, 2

let creates a new binding for each iteration.

Hoisting with Closures‚Äã

Problem 11: var in Closure‚Äã

function createFunctions() {
  var result = [];
  for (var i = 0; i < 3; i++) {
    result.push(function() {
      return i;
    });
  }
  return result;
}

const funcs = createFunctions();
console.log(funcs[0]()); // 3
console.log(funcs[1]()); // 3
console.log(funcs[2]()); // 3

Explanation: All closures reference the same i variable.

Fix:

function createFunctions() {
  var result = [];
  for (let i = 0; i < 3; i++) {
    result.push(function() {
      return i;
    });
  }
  return result;
}

const funcs = createFunctions();
console.log(funcs[0]()); // 0
console.log(funcs[1]()); // 1
console.log(funcs[2]()); // 2

Problem 12: Closure with setTimeout‚Äã

for (var i = 1; i <= 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, i * 1000);
}

Output: 4, 4, 4 (after 1s, 2s, 3s)

Fix with IIFE:

for (var i = 1; i <= 3; i++) {
  (function(j) {
    setTimeout(function() {
      console.log(j);
    }, j * 1000);
  })(i);
}
// Output: 1, 2, 3

Fix with let:

for (let i = 1; i <= 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, i * 1000);
}
// Output: 1, 2, 3

Advanced Collision Scenarios‚Äã

Problem 20: Function vs var Collision‚Äã

console.log(foo); // [Function: foo]

var foo = "variable";
function foo() {
  return "function";
}

console.log(foo); // "variable"

Explanation:

// Creation phase:
function foo() { return "function"; } // Function hoisted
var foo; // var declaration (but doesn't override the function)

// Execution phase:
console.log(foo); // [Function: foo]
foo = "variable"; // Assignment happens
console.log(foo); // "variable"

Problem 21: Multiple Functions Same Name‚Äã

foo(); // "third"

function foo() {
  console.log("first");
}

function foo() {
  console.log("second");
}

function foo() {
  console.log("third");
}

Explanation: Last function declaration wins.

Problem 22: Parameter vs var‚Äã

function test(x) {
  console.log(x); // 10
  var x;
  console.log(x); // 10
  x = 20;
  console.log(x); // 20
}
test(10);

Explanation: Parameter declaration takes precedence. var x; is redundant.

Problem 23: let in Parameter Default‚Äã

function test(a = b, b = 2) {
  console.log(a, b);
}
test(); // ‚ùå ReferenceError: Cannot access 'b' before initialization

Explanation: Parameters are evaluated left to right. b is in TDZ when a's default is evaluated.

Fix:

function test(b = 2, a = b) {
  console.log(a, b);
}
test(); // 2, 2

Common Production Bugs‚Äã

Bug 1: Conditional var Declaration‚Äã

‚ùå Buggy:

function checkStatus() {
  if (!isReady) {
    var isReady = true;
  }
  return isReady;
}
console.log(checkStatus()); // undefined (not true!)

Behind the scenes:

function checkStatus() {
  var isReady; // Hoisted to top, initialized as undefined
  if (!isReady) { // undefined is falsy
    isReady = true;
  }
  return isReady;
}

‚úÖ Fix:

function checkStatus() {
  let isReady = false;
  if (!isReady) {
    isReady = true;
  }
  return isReady;
}
console.log(checkStatus()); // true

Bug 2: Loop Event Handlers‚Äã

‚ùå Buggy:

for (var i = 0; i < buttons.length; i++) {
  buttons[i].onclick = function() {
    alert(i); // Always alerts buttons.length
  };
}

‚úÖ Fix:

for (let i = 0; i < buttons.length; i++) {
  buttons[i].onclick = function() {
    alert(i); // Alerts correct index
  };
}

Bug 3: Async var Mutation‚Äã

‚ùå Buggy:

for (var i = 0; i < 5; i++) {
  fetch('/api/' + i).then(response => {
    console.log('Response for', i); // Always 5
  });
}

‚úÖ Fix:

for (let i = 0; i < 5; i++) {
  fetch('/api/' + i).then(response => {
    console.log('Response for', i); // Correct index
  });
}

Mental Model for Interviews‚Äã

When explaining hoisting in interviews, use this mental model:

"JavaScript processes code in two phases. During the creation phase, it scans through the scope and registers all variable and function declarations, allocating memory and initializing them according to specific rules‚Äîvar gets undefined, functions get their full definition, and let/const/classes enter a temporal dead zone where they exist but can't be accessed. Then during the execution phase, the code runs line by line with those pre-registered bindings already in place. This is why you can call a function before it's declared, but accessing a let variable before declaration throws a ReferenceError."

Key phrases to use:

  • "Two-phase mechanism: creation and execution"
  • "Memory allocation vs initialization"
  • "Temporal Dead Zone for let/const"
  • "Scope registration before execution"
  • "Function declarations are fully initialized"

Interview Checklist‚Äã

‚úÖ Function declarations ‚Üí Fully hoisted and initialized ‚úÖ var ‚Üí Hoisted, initialized to undefined ‚úÖ let/const/class ‚Üí Hoisted but in TDZ, uninitialized ‚úÖ Arrow functions ‚Üí Not hoisted as functions ‚úÖ Function expressions ‚Üí Variable hoisted (if var), function not ‚úÖ Parameters ‚Üí Take precedence over var in same scope ‚úÖ Closures + var ‚Üí Single shared binding (common bug) ‚úÖ Block scope ‚Üí let/const respect blocks, var doesn't ‚úÖ typeof ‚Üí Doesn't bypass TDZ ‚úÖ Switch cases ‚Üí Share one block scope

Ultra-Hard Puzzles‚Äã

üî• Puzzle 1: Complex Shadowing‚Äã

var a = 1;

function outer() {
  console.log(a);

  function inner() {
    console.log(a);
    var a = 3;
  }

  inner();
  console.log(a);
  var a = 2;
}

outer();
console.log(a);

Output:

undefined
undefined
undefined
1

Explanation:

  • First console.log(a) in outer: local var a is hoisted ‚Üí undefined
  • In inner, var a is hoisted ‚Üí undefined
  • After inner(), still undefined in outer
  • After outer(), global a is still 1

üî• Puzzle 2: Function Expression Collision‚Äã

var foo = 1;

function bar() {
  console.log(foo);
  foo = 10;

  function foo() {}

  console.log(foo);
}

bar();
console.log(foo);

Output:

[Function: foo]
10
1

Explanation:

  • Function foo is hoisted in bar's scope
  • foo = 10 reassigns the local function
  • Global foo remains 1

üî• Puzzle 3: TDZ with Function Call‚Äã

let x = 1;

function test() {
  console.log(x);
  let x = 2;
}

test();

Output: ReferenceError: Cannot access 'x' before initialization

Explanation: Local let x creates TDZ in test, shadowing outer x.

üî• Puzzle 4: Multiple Scopes‚Äã

var x = 1;

{
  var x = 2;
  {
    let x = 3;
    console.log(x);
  }
  console.log(x);
}

console.log(x);

Output:

3
2
2

Explanation: var ignores block scope, let respects it.

üî• Puzzle 5: Async + Hoisting‚Äã

for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    var j = i;
    console.log(j);
  }, 100);
}

Output: 3, 3, 3 (after 100ms)

Explanation: var i is shared. Each timeout captures the final value of i.

üî• Puzzle 6: Class Expression‚Äã

const MyClass = class InternalName {
  constructor() {
    console.log(InternalName);
  }
};

new MyClass(); // [class InternalName]
console.log(InternalName); // ‚ùå ReferenceError

Explanation: Class expression name is only visible inside the class.

üî• Puzzle 7: Destructuring Hoisting‚Äã

console.log(a); // ‚ùå ReferenceError
let { a } = { a: 1 };
console.log(b); // undefined
var { b } = { b: 2 };

Explanation: Destructuring follows the same hoisting rules as regular declarations.

üî• Puzzle 8: Function Parameter Default with Closure‚Äã

var x = 1;

function foo(x = x) {
  console.log(x);
}

foo(); // ‚ùå ReferenceError: Cannot access 'x' before initialization

Explanation: Parameter x is in TDZ when its own default value is evaluated.

üî• Puzzle 9: Nested Function Hoisting‚Äã

function outer() {
  inner(); // ‚úÖ Works

  function inner() {
    console.log("inner");
  }
}

outer();
function outer() {
  inner(); // ‚ùå ReferenceError

  const inner = function() {
    console.log("inner");
  };
}

outer();

üî• Puzzle 10: Mixed Declarations‚Äã

console.log(typeof foo); // "function"
console.log(typeof bar); // "undefined"

var bar = function() {};

function foo() {}

Explanation: Function declaration hoisted completely, function expression only hoists the variable.

üî• Puzzle 11: with Statement (Don't use in production!)‚Äã

var obj = { a: 1 };

with (obj) {
  console.log(a); // 1
  var a = 2;      // Creates global variable!
}

console.log(a);     // 2
console.log(obj.a); // 1

Explanation: var inside with doesn't create a property on obj, it creates a global variable.

üî• Puzzle 12: eval and Hoisting (Avoid eval!)‚Äã

function test() {
  eval("var x = 10");
  console.log(x); // 10 (in non-strict mode)
}

test();

In strict mode:

"use strict";
function test() {
  eval("var x = 10");
  console.log(x); // ‚ùå ReferenceError
}

test();

üî• Puzzle 13: Generator Function‚Äã

gen(); // ‚ùå ReferenceError

const gen = function* () {
  yield 1;
};
gen(); // ‚úÖ Works

function* gen() {
  yield 1;
}

Explanation: Generator expressions follow function expression rules, generator declarations follow function declaration rules.

üî• Puzzle 14: Arrow Function with var‚Äã
